# report スキル

レポートとは、作業の内容（何を作業したか、どのように作業したか、作業中に気づいたことや考えたことなど）をまとめたものです。報告書とも呼ばれます。

プロジェクトのトップから見て `./report/YYYYMMDD.md` という形式でレポートを保存します。例えば、2024年6月1日に作業した内容をレポートする場合は、`./report/20240601.md` というファイルにレポートを保存します。

`./report/members.md` というファイルには、プロジェクトのメンバーの名前と役割を記載します。例えば、以下のような内容になります。

```
# メンバー
- Alice: Surveyor -- 論文調査担当
- Bob: Researcher -- 実装担当
- Charlie: Engineer -- システム構築担当
```

## /report init

1. `./report/` ディレクトリが存在しない場合は、ディレクトリを作成する
2. `./report/members.md` というファイルが存在しない場合は、ファイルを作成する

---

## /report write <報告者名>

1. 作業内容の見出しと本文（箇条書き）を決める
2. メンバーリスト (`./report/members.md`) に報告者名が存在するか確認する
    - 存在しない場合は、ユーザーに報告者名と役割をヒアリングしてメンバーリストに追記する
3. 以下のスクリプトを実行してレポートを追記する（時刻はスクリプト内で自動取得）

```bash
bash ${CLAUDE_SKILL_DIR}/scripts/write.sh "<報告者名>" <<EOF
### <作業内容の見出し>
- <報告内容>
    - <何を作業した>
    - <この問題で詰まったがこうやって解決した>
    - <誰々さんのこの作業のせいでこうなったので次からは報告してほしい>
EOF
```

詳細は `--help` オプションを参照してください。

---

## /report read <YYYYMMDD>

1. `./report/YYYYMMDD.md` というファイルが存在するか確認する
2. あれば Read する

## レポートの形式

一つのレポートは以下のような形式で記述します。

```
# YYYY-MM-DD

## HH:MM - (報告者の名前)

### 作業内容の見出し

- 報告内容
    - 何を作業した
    - この問題で詰まったがこうやって解決した
    - 誰々さんのこの作業のせいでこうなったので次からは報告してほしい
```

報告者名はあなたの「役割」としての名前が適切です。
例えば "Surveyor" や "Researcher" など、あなたがその日に行った作業の内容を表す名前が良いでしょう。

この報告は将来の自分自身とプロジェクトの他のメンバーが、過去の作業内容を振り返るために使用されます。したがって、できるだけ具体的で詳細な内容を記述することが望ましいです。また他メンバーへの依頼も書いておくとよいかもしれません。

以下はレポートの例です。

```
# 2026-03-13

## 14:30 - Surveyor

### SASRec (Self-Attentive Sequential Recommendation) 論文調査

#### 作業内容

- arxiv で古典かつ優秀な推薦手法を検索し、SASRec (1808.09781) を選定
- README.md の調査済み手法テーブルに SASRec を追加 (Implemented: ❌️)
- arxiv-mcp-server で論文をダウンロード・読解
- `papers/1808.09781.SASRec_Sequential_Attention.md` としてサマリドキュメントを作成

#### 選定理由

- 2018年発表のTransformer-based逐次推薦の先駆け的論文で、多数の後続研究の基盤となっている古典的手法
- データセットに timestamp があるため逐次推薦との相性が良い
- スパース・デンス両方のデータセットで既存手法（MC/RNN/CNN系）を上回る精度
- CNN/RNN系の10〜18倍の学習速度という実用的な優位性
- RecBole に既存実装があり、実装コストが低い

#### 論文のポイント

- Self-Attention + 因果マスクにより、未来情報の漏洩を防ぎつつ過去の全系列を参照
- Shared Item Embedding が性能向上に必須（非共有では大幅に性能低下）
- Residual Connection + Layer Normalization + Dropout の組み合わせが安定学習に重要
- スパースデータでは直近アイテムに集中、デンスデータでは広い履歴を参照する適応的 Attention

#### 気づいたこと

- このリポジトリのデータ (user_id, content_id, timestamp) は SASRec の入力形式にそのまま対応する
- RecBole でラップすれば実装が比較的容易
- BERT4Rec など後続の逐次推薦手法への橋渡しとして、まず SASRec を実装・評価することに価値がある
```

## まとめ

- `/report` スキルは、作業内容のレポートを管理するためのスキルです
- `/report init`: レポートディレクトリとメンバーファイルの初期化
- `/report write <報告者名>`: レポートの追記
- `/report read <YYYYMMDD>`: レポートの読み込み
